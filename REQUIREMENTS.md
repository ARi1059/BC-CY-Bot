# BC-CY-Bot 需求文档

## 1. 项目概述

### 1.1 项目名称
BC-CY-Bot —— 一次性入群邀请链接审核机器人

### 1.2 项目目标
开发一个 Telegram Bot，用于管理用户的入群申请。用户通过 Bot 提交申请材料，由对应的"邀请人"审核（部分邀请人采用代审模式，由管理员统一审核），审核通过后由 Bot 自动签发**一次性入群邀请链接**（member_limit=1），并将链接使用情况同步到日志频道。

### 1.3 核心价值
- 将传统的人工拉群流程标准化、自动化
- 全程**内联按钮（Inline Keyboard）引导式操作**，零命令行学习成本
- 邀请链接一次性失效，杜绝滥用
- 链接使用全程留痕，日志频道实时可查（谁邀请了谁）
- **出击报告自动归档**：审核通过后自动转发到指定频道，便于战绩沉淀与回溯
- **回群密钥救济通道**：原账号丢失/封禁后，凭密钥可换取新一次性链接，无需重走审核
- 支持多组别、多群组、多管理员的分层管理

## 2. 角色定义

| 角色 | 说明 | 权限 |
|------|------|------|
| **申请人 (User)** | 希望加入群组的普通 Telegram 用户 | 发起申请、选择邀请人、提交材料、接收审核结果与邀请链接 |
| **邀请人 (Inviter)** | 由管理员预先配置的审核者，对应一个或多个组别。分两种**审核模式**：**自审型 / 代审型** | 自审型：审核归属自己的申请；代审型：仅作为"挂名邀请人"展示，实际审核由管理员统一处理。**注：审核模式仅在邀请人管理后台可见，对申请人侧与日志频道均不暴露** |
| **超级管理员 (Super Admin)** | 主管理员，**全系统唯一一位**（初始化时指定，可转让） | 拥有所有副管理员权限 + **独享**：任命/卸任副管理员、转让超级管理员身份 |
| **副管理员 (Sub Admin)** | 由超级管理员任命，**可有多个** | 与超级管理员同等的业务操作权限（群组/邀请人/黑名单/日志频道/代审审核/统计），**但无任何针对管理员的任命与卸任权限**（不可增、不可减、不可改其他管理员） |

> - 一个 Telegram 用户可以同时担任多个角色（如同时是邀请人和管理员，角色不互斥）。
> - 主/副管理员在业务审核与配置层面**完全平权**，对同一申请任一管理员先操作即生效。
> - 主/副差异**仅体现在「管理员管理」这一个子模块**：副管理员看不到该模块的写操作按钮（只读列表）。

### 2.1 交互设计原则

为降低使用门槛，本 Bot 遵循以下交互规范：

1. **全程 Inline Keyboard 操作**：所有非首次启动的操作通过内联按钮完成，用户/邀请人/管理员**无需记忆任何命令**。
2. **唯一命令入口**：
   - `/start` —— 申请人入口（弹出申请引导面板）
   - `/admin` —— 管理员入口（弹出管理面板）
   - `/panel` —— 邀请人入口（弹出审核/统计面板）
   - `/help` —— 帮助信息
3. **引导式（Wizard）流程**：每一步都明确告诉用户「当前在做什么」「下一步是什么」「随时可返回/取消」。每条指令性消息都包含进度提示，例如 `(2/3) 请上传上课手势`。
4. **按钮分组**：超过 4 个的按钮按 2 列网格排布；列表型数据（如多个邀请人）使用按钮分页（每页 6 条 + 上一页/下一页）。
5. **可逆操作**：所有引导步骤提供 `« 上一步` / `❌ 取消` 按钮；管理面板的破坏性操作（删除邀请人、清除黑名单等）需二次确认。

#### 3.1.3 申请人辅助操作（均通过按钮，不依赖命令）
- 申请进行中：`📋 查看进度` / `❌ 取消申请` / `« 上一步`
- 已提交后：`📋 查看进度` / `🔄 重新申请（取消当前）`
- 错误兜底：任何意外输入 → Bot 回复"请使用下方按钮操作" + 重新显示当前步骤按钮

### 3.2 邀请人侧（审核者）

#### 3.2.1 审核消息推送（双消息模式：媒体组+报告 / 申请人信息+按钮）

> **重要**：Telegram Bot API `sendMediaGroup` 不支持 `reply_markup`（无法附带按钮）。本 Bot 采用**两条消息**结构：
> - **消息 ①**：媒体组（约课记录 + 上课手势两张图片合为相册），**出击报告作为首图 caption**
> - **消息 ②**：申请人元信息 + 审核按钮

**收到归属于自己的新申请时，Bot 私聊推送给邀请人（或所有管理员，代审型时）：**

```
─── 消息 ① （媒体组 + 出击报告 caption）───
[图片1: 约课记录]    ← caption 位置
[图片2: 上课手势]
caption: <出击报告全文>
（两张图片由 sendMediaGroup 组成相册，首图 caption 承载出击报告）

─── 消息 ② （申请人信息 + 审核按钮）───
📥 新申请待审核 #A1024
─────────────────────────
👤 申请人：@username
🆔 ID：123456789
🎓 邀请人：张老师（A组）
🕐 提交时间：2026-05-12 14:23
─────────────────────────
[✅ 通过]   [❌ 拒绝]
[👁 重发审核材料]
```

**关键规则：**
- 出击报告**不再重复出现**在消息 ② 中，统一作为消息 ① 的 caption
- 媒体组 caption 限长 **1024 字符**；若出击报告超长，降级为**三条消息**模式：
  ```
  ① 媒体组（无 caption）
  ② 出击报告全文（独立文本消息）
  ③ 申请人信息 + 审核按钮
  ```
- 若组别不含图片材料，则只发送消息 ②（含出击报告 + 按钮）
- `[👁 重发审核材料]`：重新推送本申请的完整材料（防止被新消息淹没）

> **为什么必须双消息？** 见上方限制说明：Telegram 不允许在媒体组上附带 inline keyboard，所以审核按钮必须单独走文本消息。这也是 §3.1.1 强制申请人逐项提交、禁用媒体组的前置依据 —— 服务端始终按"单张图片 × N"持久化，发审核时统一拼成媒体组即可。

#### 3.2.2 审核操作

- **点击「✅ 通过」**：
  1. Bot 调用 `createChatInviteLink` 生成一次性链接
  2. **生成回群密钥**（仅首次审核通过时生成，重新申请的不重复生成；见 §3.8）
  3. 私聊推送链接 + 回群密钥给申请人
  4. 在审核消息原位置编辑为"✅ 已通过 by @审核人 · 14:25"
  5. 自动向**日志频道**推送审核结果（见 §3.6）
  6. **若该申请包含「出击报告」材料**，自动转发到**出击报告频道**（见 §3.7）

- **点击「❌ 拒绝」**：
  1. Bot 询问"是否填写拒绝原因？"
       `[✏️ 填写原因]   [⏩ 跳过直接拒绝]`
  2. 选择「填写原因」→ Bot 等待邀请人下一条文本消息 → 回写至 application
  3. 选择「跳过」→ 直接拒绝（原因留空）
  4. 通知申请人；审核消息编辑为"❌ 已拒绝 by @审核人 · 14:25"
  5. 同步日志频道

#### 3.2.3 邀请人面板 `/panel`

通过 `/panel` 命令打开个人审核面板：

```
🧑‍🏫 邀请人面板 · 张老师

📥 待审核：3 条
📊 我的统计

[📥 待审核列表 (3)]
[📊 我的统计]
[❓ 帮助]
```

- **待审核列表**：分页展示自己名下所有 `pending` 申请，点击进入审核
- **我的统计**：本月/累计审核数、通过数、拒绝数、链接使用率

### 3.3 管理员侧

#### 3.3.1 入口：`/admin`

```
🛠 管理面板

[👥 群组管理]      [🎓 邀请人管理]
[🚫 黑名单管理]    [👮 管理员管理]
[📡 日志频道]      [📋 出击报告频道]
[📥 待我审核 (N)]  [📊 全局统计]
[🔑 回群密钥管理]  [⚙️ 系统配置]*
[❓ 帮助]

*`[⚙️ 系统配置]` 仅超级管理员可见
```

> 所有子菜单均为按钮交互，**无需记忆命令**。下文括号内的命令为可选的快捷入口（高级用户使用）。

#### 3.3.2 群组管理 `[👥 群组管理]`
- 列表展示已纳管群组（每行一条 + 「移除」按钮）
- `[➕ 添加群组]` → Bot 提示"请将我加入目标群并设为管理员，然后转发群内任一消息到此处" → 自动识别并入库

#### 3.3.3 邀请人管理 `[🎓 邀请人管理]`
- 分页列表展示所有邀请人，点击进入详情
- 详情页：编辑昵称、组别、关联群组、所需材料、**审核模式**（`自审 / 代审`）、启用/停用
- `[➕ 添加邀请人]` 引导流程：
  1. 转发该邀请人发给 Bot 的任意消息 / 输入 Telegram 用户 ID
  2. 输入组别名（如 "A组"）
  3. 选择关联群组（按钮列表）
  4. 选择所需材料（多选复选框：约课记录 / 上课手势 / 出击报告）
  5. 选择审核模式：`👤 自审型（邀请人本人审核）` / `🏢 代审型（由管理员统一审核）`
  6. 确认入库

#### 3.3.4 黑名单管理 `[🚫 黑名单管理]`
- 列表 + 「解除」按钮
- `[➕ 添加黑名单]` → 输入用户 ID 或转发该用户消息 → 可选填原因 → 确认

#### 3.3.5 管理员管理 `[👮 管理员管理]`

**权限分层（仅本模块生效，其他业务模块主/副平权）：**

| 操作 | 超级管理员 | 副管理员 |
|------|:---------:|:-------:|
| 查看管理员列表 | ✅ | ✅（只读） |
| 添加副管理员 | ✅ | ❌ |
| 移除副管理员 | ✅ | ❌ |
| 转让超级管理员身份 | ✅ | ❌ |
| 移除超级管理员 | ❌（无此操作） | ❌ |

**界面表现：**

- **超级管理员视角**：
  ```
  👮 管理员管理

  👑 超级管理员
   └ @superadmin (123456789)  [🔄 转让身份]

  🛡 副管理员 (2)
   └ @sub1 (234567890)  [➖ 移除]
   └ @sub2 (345678901)  [➖ 移除]

  [➕ 添加副管理员]
  ```
- **副管理员视角**：
  ```
  👮 管理员列表（只读）

  👑 超级管理员
   └ @superadmin

  🛡 副管理员 (2)
   └ @sub1 （我）
   └ @sub2

  ⓘ 仅超级管理员可任命或卸任副管理员
  ```

**关键规则：**
- 超级管理员**全系统唯一**；不可被任何管理员直接移除
- `[🔄 转让身份]`：超级管理员可将身份转交给任一现有副管理员，自身降级为副管理员（二次确认 + 输入"确认转让"防误触）
- `[➕ 添加副管理员]`：输入用户 ID 或转发其任意消息 → 二次确认 → 入库
- `[➖ 移除副管理员]`：每个副管理员行末按钮，二次确认
- 任何非超级管理员尝试调用写操作（哪怕通过伪造 callback_data）→ 服务端校验拦截 + 写审计日志

#### 3.3.6 日志频道设置 `[📡 日志频道]`
- 显示当前绑定的日志频道（或"未设置"）
- `[➕ 绑定频道]`：提示"请将 Bot 加入频道并设为管理员，然后转发频道任一消息到此处" → 自动绑定
- `[🔄 更换频道]` / `[➖ 解绑]`
- 详情见 §3.6

#### 3.3.7 出击报告频道设置 `[📋 出击报告频道]`
- 显示当前绑定的出击报告频道（或"未设置"）
- `[➕ 绑定频道]`：流程同日志频道（Bot 加入频道并设为管理员 → 转发频道任一消息到此处自动识别）
- `[🔄 更换频道]` / `[➖ 解绑]`
- `[👁 预览最近 5 条转发记录]`：便于核对频道是否生效
- 详情见 §3.7

#### 3.3.8 待我审核 `[📥 待我审核 (N)]`
- 用于代审型邀请人申请的统一审核入口
- 多个管理员共享同一队列，**先操作者锁定该申请**（避免重复审核）
- 审核消息推送规则同 §3.2.1（媒体组 + 文本+按钮两条消息），但同时推送给所有管理员（任一管理员处理即标记完成，其他管理员的消息按钮自动失效）

#### 3.3.9 全局统计 `[📊 全局统计]`
- 总申请数、通过率、拒绝率
- 各邀请人 / 组别的通过率
- 链接签发数 vs 实际使用数
- 回群密钥签发数 / 已使用数 / 已撤销数
- 异常事件数（链接被非申请人使用、超期未用、API 失败、出击报告转发失败、密钥被同 ID 使用拦截等）

#### 3.3.10 回群密钥管理 `[🔑 回群密钥管理]`
- 默认展示「最近 20 条密钥使用记录」+ 搜索框
- `[🔍 按申请 ID 查询]` / `[🔍 按 Telegram ID 查询]`
- 单条密钥详情页：
  ```
  🔑 密钥 #K2031（关联申请 #A1024）
  ─────────────────────────
  原始持有人：@user1 (123456789)
  当前持有人：@user2 (987654321)
  密钥前缀：BCCY-A7K9-****
  状态：✅ 活跃
  生成时间：2026-04-20 10:00
  最近一次使用：2026-05-10 22:13（由 @user2）

  [♻️ 重置密钥]   [🚫 撤销密钥]   [📜 完整使用历史]
  ```
- `[♻️ 重置密钥]`：作废现有密钥，生成新密钥并私聊推送给当前持有人（需输入"确认重置"防误触）
- `[🚫 撤销密钥]`：永久作废，不再可用（如怀疑密钥泄露）
- 详情见 §3.8

#### 3.3.11 系统配置 `[⚙️ 系统配置]`（仅超级管理员可见）

副管理员**不可见此模块**，UI 层不渲染按钮 + 服务端 callback 校验拦截。

```
⚙️ 系统配置

📌 邀请链接配置
 ├ 有效期：24 小时   [✏️ 修改]
 └ 单次可用：1 次（固定，不可修改）

📌 其他
 └ （预留：未来扩展项）

[« 返回管理面板]
```

- `[✏️ 修改有效期]`：输入数字（1–168 小时之间），二次确认后写入 settings 表，立即生效（仅影响修改后新签发的链接，存量链接不变）
- 修改记录写入 audit_logs（actor / 旧值 / 新值 / 时间）

### 3.4 邀请链接生成规则

调用 Telegram Bot API `createChatInviteLink`：

| 参数 | 取值 | 说明 |
|------|------|------|
| `chat_id` | 对应组别的目标群组 ID | 决定拉进哪个群 |
| `member_limit` | `1`（固定不可配） | **一次性使用，硬约束** |
| `expire_date` | 当前时间 + `invite_link_ttl_hours` 配置值 | 默认 24 小时；**仅超级管理员可在 `[⚙️ 系统配置]` 修改** |
| `name` | `App-{application_id}` | 便于回查 |
| `creates_join_request` | `false`（默认） | 已经过 Bot 审核，无需群主二次审核 |

### 3.5 链接使用追踪

- Bot 持续监听 `chat_member` 事件
- 当有用户通过邀请链接入群时，记录：
  - 申请 ID
  - 实际入群的 Telegram 用户 ID 和用户名
  - 入群时间
- 若实际入群者与申请人不一致：标记 `anomaly=true`，向日志频道推送告警 + @所有管理员

### 3.6 日志频道（邀请追踪日志）

**目的**：让所有管理员/股东可一目了然地查看"谁邀请了谁"，便于审计与对账。

#### 3.6.1 频道形态
- 一个 Telegram 频道（Channel），Bot 必须是该频道的管理员
- 频道只读，Bot 自动推送结构化日志卡片
- 管理员可邀请相关人员加入频道作为只读观察者

#### 3.6.2 推送事件类型

| 事件 | 触发时机 | 卡片内容 |
|------|---------|---------|
| **📥 新申请** | 用户完成材料提交 | 申请ID / 申请人 / 邀请人 / 组别 / 时间 |
| **✅ 审核通过** | 审核人点通过 | 申请ID / 申请人 / 邀请人 / 审核人（自审/管理员） / 时间 / 邀请链接预览（已脱敏） |
| **❌ 审核拒绝** | 审核人点拒绝 | 申请ID / 申请人 / 邀请人 / 审核人 / 拒绝原因 / 时间 |
| **🚪 链接已使用** | 监听到 chat_member 入群事件 | 申请ID / 申请人 / 邀请人 / 实际入群人 / 入群时间 / 是否一致 |
| **⚠️ 异常告警** | 链接被非申请人使用 / API 失败 / 过期未用 | 申请ID / 异常类型 / 建议处理 |

#### 3.6.3 卡片示例

```

🎓 邀请人：王老师（B组）
👮 审核人：管理员 @admin1

```

```

👤 申请人：@apply_user (123456789)
🎓 邀请人：王老师（B组）
🚪 实际入群人：@apply_user (123456789)  ✓ 一致
🕐 入群时间：2026-05-12 14:32
```

```
⚠️ 异常告警 #A1024
─────────────────────────

实际入群人：@stranger (987654321)  ✗ 不一致！
```

> **说明**：日志频道卡片仅展示邀请人昵称与组别，**不暴露其审核模式（自审/代审）**。是否为代审场景可通过「审核人」字段间接体现（自审型由邀请人本人审核；代审型由某位管理员审核）。

### 3.7 出击报告频道（战绩归档）

**目的**：将所有审核通过的申请中的「出击报告」材料自动转发到指定频道，形成可追溯的战绩档案。

#### 3.7.1 频道形态
- 一个独立 Telegram 频道（Channel），与日志频道**分开存放**（避免出击报告与运营事件混杂）
- Bot 必须是该频道的管理员
- 频道只读，仅 Bot 推送内容
- 管理员可邀请相关人员加入频道作为只读观察者

#### 3.7.2 触发时机
- **仅在审核通过时触发**（拒绝/取消的申请不转发）
- 若申请未配置「出击报告」材料项 → 静默跳过，不视为错误
- 若频道未绑定 → 静默跳过 + 在管理员面板顶部提示「⚠️ 出击报告频道未配置」

#### 3.7.3 转发实现（仅转发出击报告原始消息）

不构造卡片、**仅转发"出击报告"这一条消息**（不转发约课记录、上课手势两张图片），避免老师个人信息通过图片泄漏到出击报告频道观察者侧。

直接调用 Telegram Bot API `forwardMessage`：

| 参数 | 取值 |
|------|------|
| `chat_id` | 出击报告频道 ID |
| `from_chat_id` | Bot 与申请人的私聊 ID（即申请人的 Telegram ID） |
| `message_id` | **仅** `application_materials` 表中 `material_type='出击报告'` 那条记录的 `original_message_id` |
| `disable_notification` | `true`（静默推送，避免打扰频道观察者） |

**严格边界：**
- ❌ 不转发约课记录图片
- ❌ 不转发上课手势图片
- ❌ 不附加任何申请 ID / 邀请人 / 申请人元信息卡片
- ✅ 仅一条出击报告原始消息（文本）通过 forwardMessage 进入频道
- 频道内呈现为 Telegram 标准 "Forwarded from <用户>" 格式
- 与申请 ID 的关联通过 `attack_report_forwards` 表回查，不在频道消息中显式标注

#### 3.7.4 与日志频道的关系

| 维度 | 日志频道 | 出击报告频道 |
|------|---------|------------|
| **定位** | 运营审计（谁邀请了谁、链接是否被使用） | 战绩归档（沉淀出击报告内容） |
| **推送事件** | 5 类（新申请/通过/拒绝/已使用/异常） | 仅 1 类（审核通过且包含出击报告） |
| **隐私敏感度** | 高（含 Telegram ID、链接） | 中（仅昵称 + 报告内容） |
| **观察者群体** | 管理员/股东 | 战绩公示相关人员 |
| **必须配置？** | 否（不影响主流程） | 否（不影响主流程） |

### 3.8 回群密钥（账号丢失救济通道）

**目的**：解决用户原 Telegram 账号丢失/被封禁后，无法重新进入群组的问题。允许用户用**新账号**凭预先持有的密钥换取一次性入群链接，无需重走审核流程。

#### 3.8.1 设计目标

- 让已通过审核的用户在更换 Telegram 账号后仍能保留入群资格
- 通过**新旧账号 ID 强制不一致**校验防止密钥被滥用、转卖、共享
- 全程可审计，密钥使用事件同步日志频道

#### 3.8.2 密钥签发

- **签发时机**：申请首次审核通过时自动生成（与入群链接同时发送）
- **签发对象**：审核通过申请的原始申请人
- **格式**：`BCCY-XXXX-XXXX-XXXX-XXXX`（4 段 × 4 位 Base32 字符，去除易混字符 `0/O/1/I/L`，共 16 位有效字符），整体熵约 80 bits
- **唯一性**：全系统唯一，碰撞时重生成
- **存储**：服务端仅存哈希（Argon2id），明文仅在生成瞬间发送给用户一次
- **有效期**：**永久有效**（仅通过"已使用 / 撤销 / 重置"流转状态，不设到期时间）
- **使用次数**：每把密钥**仅可成功使用 1 次**，使用后立即作废并为新持有人签发新密钥（链式流转）
- **每个申请仅一把当前有效密钥**：重新审核同一申请人时不重复签发；如需更换需走「重置密钥」流程
- **目标群组**：密钥使用时走**当前 inviter 配置的 target_group_id**（与重新申请逻辑一致），不锁定签发时的群

#### 3.8.3 使用流程（用户侧）

```
┌──────────────────────────────────────────┐
│  Step 0  /start                          │
│  [🚀 开始申请入群]                        │
│  [🔑 我有回群密钥]   ← 点击进入回群流程     │
└──────────────────────────────────────────┘
                ↓
┌──────────────────────────────────────────┐
│  Step 1  Bot 引导输入密钥                 │
│  "请粘贴您的回群密钥：                     │
│   格式 BCCY-XXXX-XXXX-XXXX-XXXX"          │
│         [« 返回]                          │
└──────────────────────────────────────────┘
                ↓
            校验密钥
                ↓
   ┌─────────────┴──────────────┐
   ↓                            ↓
 校验失败                     校验通过
   │                            │
   ▼                            ▼
┌──────────────┐    ┌────────────────────────┐
│ 失败原因分类  │    │ 生成新一次性入群链接     │
│ + 友好提示    │    │ + 同时签发新回群密钥     │
│ + 失败计数+1  │    │ + 旧密钥状态置 'used'   │
└──────────────┘    │ + 推送日志频道           │
                    └────────────────────────┘
```

#### 3.8.4 校验规则（按顺序执行，任一失败即中止）

| 顺序 | 校验项 | 失败处理 |
|:---:|------|---------|
| 1 | 输入格式符合 `BCCY-XXXX-XXXX-XXXX-XXXX` | 「密钥格式不正确」 |
| 2 | 密钥哈希在 recovery_keys 表中存在且 status='active' | 「密钥无效、已使用或已撤销」（错误信息合并不暴露原因，防探测） |
| 3 | **当前请求者的 Telegram 数字 ID ≠ 密钥的 owner_telegram_id**（★ 核心防滥用） | 「该密钥仅用于在更换账号时使用，您当前账号与原账号一致，无需使用密钥。如需重新生成请联系管理员」 |
| 4 | 当前请求者不在黑名单 | 「您已被列入黑名单」 |
| 5 | 密钥对应的 inviter 仍处于 active 状态 | 「关联邀请人已停用，请联系管理员」 |
| 6 | 频率限制：同一密钥近 1 小时失败尝试 < 5 次 | 「尝试次数过多，请 1 小时后再试 / 联系管理员重置」 |
| 7 | 频率限制：同一新 Telegram ID 近 24 小时使用密钥 < 3 次 | 「该账号 24 小时内回群次数过多」 |

校验全通过后：
1. 调用 `createChatInviteLink` 生成一次性链接（target_group_id 取**当前** inviter 配置）
2. 旧密钥状态置 `'used'`，记录 `used_by_telegram_id` 与 `used_at`
3. **自动签发新密钥给新账号**（current_owner 更新为新 ID），便于新账号将来再次救济
4. **对原账号执行清理动作**（见 §3.8.5，根据原账号状态自动选择踢出或封禁）
5. 推送日志频道（事件类型：🔑 密钥使用，含清理动作结果）
6. 应用表 `applications.applicant_telegram_id` **不被覆盖**，原始 ID 保留作为审计基线；新 ID 仅记录在 recovery_keys 链上

#### 3.8.5 原账号清理（密钥使用后自动执行）

为防止"原账号 + 新账号"同时占用群组名额、或注销账号 ID 将来被复用造成隐患，密钥成功使用后 Bot 立即对**原账号 ID**（即 `owner_telegram_id`）执行清理：

**清理决策表：**

| 原账号状态 | 是否在目标群内 | 清理动作 | 说明 |
|-----------|:-------------:|---------|------|
| **正常账号**（未注销） | ✅ 在群内 | **踢出本群**（`banChatMember` → 立即 `unbanChatMember`） | 释放群组名额；用户的新账号已通过密钥入群，原账号无需保留访问 |
| **正常账号**（未注销） | ❌ 不在群内 | 无动作 | 原账号本就不在群，不做处理 |
| **已注销账号**（Telegram 官方注销） | 任意 | **永久封禁**（`banChatMember` 不解封，`until_date=0`）+ 写入本地黑名单（reason='账号已注销，密钥使用后自动封禁'） | 防止该 ID 在 Telegram 侧若发生异常复用时回流；同时本地黑名单防绕道再申请 |

**注销账号识别规则**：调用 `getChat(user_id)` 或 `getChatMember` 返回的 User 对象满足以下任一条件即视为已注销：
- `first_name` 为空字符串、`"Deleted Account"`、`"已注销账号"` 之一
- `last_name` 与 `username` 同时为空 **且** `first_name` 等于上述特征值
- API 返回 `Bad Request: chat not found` / `user is deactivated`（视为已注销）

**例外/兜底规则：**
- 若原账号在群内的身份为 `administrator` 或 `creator`：**不自动踢出**，转为通知所有管理员手动处理（防止误踢群主/管理员）
- 若 Bot 在目标群组缺少封禁/踢出权限：清理动作失败 → 写入 recovery_keys.cleanup_status='failed' → 日志频道告警 + 通知所有管理员，不阻塞密钥使用主流程
- 清理动作发生在新链接已发送给申请人之后，**与新账号入群之间不强制时序**（避免新账号入群延迟阻塞清理）

#### 3.8.6 用户主动操作（在「📋 查看我的申请记录」页面内）

- `[♻️ 重置回群密钥]`：原账号在仍可登录时可主动重置密钥（如担心旧密钥泄露）
  - 二次确认 + 提示"旧密钥将立即失效"
  - 生成新密钥并发送给当前持有人
  - **频率限制：每持有人 1 次/天**（防误触），命中限制时提示"今日已重置过密钥，请明日再试"
- `[🚫 弃用回群密钥]`：用户主动作废（如确认不再需要）

#### 3.8.7 与日志频道联动

新增日志频道事件类型「🔑 密钥使用」：

```
🔑 回群密钥使用 #K2031
─────────────────────────
关联申请：#A1024
🎓 邀请人：张老师（A组）
👤 原账号：@user1 (123456789)  · 状态：已注销
🚀 新账号：@user2 (987654321)
🔗 新链接：t.me/+xxxx****
🧹 清理动作：✅ 已永久封禁原账号
🕐 时间：2026-05-12 14:25
```

清理动作字段的可能值：
- `✅ 已踢出原账号（账号正常，曾在群内）`
- `✅ 已永久封禁原账号（账号已注销）`
- `➖ 无需清理（原账号未在群内）`
- `⚠️ 清理失败：Bot 缺少权限`（同步通知所有管理员人工处理）
- `⚠️ 跳过清理：原账号为群组管理员`（同步通知所有管理员人工确认）

异常事件（同 ID 拦截、频率限制触发、连续失败、清理失败）也推送日志频道，附「⚠️」前缀。

#### 3.8.8 安全考量

| 风险 | 缓解 |
|------|------|
| 密钥被截屏/转发 | 1) 同 ID 拦截使新拥有者无法用原账号反复刷链接；2) 频率限制；3) 日志频道实时可审；4) 管理员可一键撤销 |
| 暴力枚举密钥 | 1) 80 bits 熵 + Argon2 哈希；2) 同密钥 1 小时 5 次失败锁定；3) 同 IP/ID 24 小时 3 次上限 |
| 用户遗忘保存 | 1) 通过链接消息中明显的密钥卡片 + 复制按钮；2) 申请记录页随时可复制（仅在原账号未丢失时） |
| 一人多账号刷链接 | 同 ID 校验仅约束「与原账号不同」，无法防完全独立的两个新账号轮换；以日志频道 + 黑名单 + 管理员审计兜底 |
| 密钥链被劫持（账号 A 用密钥拿到链接后转给账号 B 入群） | chat_member 事件监听：实际入群人 ID 与本次密钥使用人 ID 不一致 → 日志频道异常告警 + 通知所有管理员 |

## 4. 非功能需求

### 4.1 技术栈推荐

| 类别 | 选型 | 理由 |
|------|------|------|
| 语言 | **Python 3.11+** | 生态完善、Telegram 库成熟 |
| Bot 框架 | **python-telegram-bot v21+ (async)** | 官方维护、异步、支持 CallbackQuery 与 ConversationHandler |
| 数据库 | **PostgreSQL 15+**（生产） / SQLite（开发） | 关系型，事务/审核记录强一致 |
| ORM | **SQLAlchemy 2.0 (async)** | 与异步框架天然契合 |
| 状态机 | **ConversationHandler** + 数据库快照 | 引导式流程的天然载体，崩溃可恢复 |
| 配置 | **pydantic-settings + .env** | 类型校验、环境隔离 |
| 日志 | **structlog** + 文件轮转 | 结构化日志，便于排查 |
| 部署 | **Docker + docker-compose** | 一键部署、可移植 |
| 进程管理 | systemd / Docker restart=always | 故障自动重启 |

### 4.2 性能要求
- 支持并发 100+ 用户同时申请（异步处理，无阻塞）
- 审核通知推送延迟 < 2 秒
- 日志频道推送延迟 < 3 秒
- 数据库查询 P95 < 200ms

### 4.3 可靠性
- 异常崩溃后能自动重启并恢复未完成的 Conversation 状态（基于数据库快照，不依赖内存）
- Telegram API 调用须带重试（指数退避，最多 3 次）
- 关键操作（生成链接、审核结果、日志推送）须落库后再回复，避免丢失
- 多管理员并发审核同一申请时，使用数据库行锁防止重复处理

### 4.4 安全性
- Bot Token 仅从环境变量读取，禁止硬编码
- 所有管理员/邀请人操作须校验 Telegram 用户 ID 白名单（来自 admins / inviters 表）
- 数据库密码、敏感配置全部走环境变量
- 申请材料只存 file_id（Telegram 侧），本地不落盘原文件
- 日志频道卡片中的邀请链接做部分脱敏

### 4.6 环境变量清单

| 变量名 | 必填 | 说明 |
|--------|:----:|------|
| `BOT_TOKEN` | ✅ | Telegram Bot Token |
| `DATABASE_URL` | ✅ | PostgreSQL 连接串 |
| `INITIAL_SUPER_ADMIN_ID` | ✅ | **首次启动时**写入 admins 表（role='super'）的 Telegram 数字 ID；后续启动若该 ID 已是超级管理员则忽略，若变更则覆盖现有超级管理员（用于应急换人，见下） |
| `LOG_LEVEL` | ⭕ | 默认 INFO |
| `TIMEZONE` | ⭕ | 默认 Asia/Shanghai，影响日志/卡片时间显示 |

**初始超级管理员产生：** 首次启动时读取 `INITIAL_SUPER_ADMIN_ID` 并写入 admins 表；之后超级管理员身份在 Bot 内通过「🔄 转让身份」流转。

**超级管理员账号丢失应急预案：**
1. SSH 登录服务器，编辑 `.env` 文件，将 `INITIAL_SUPER_ADMIN_ID` 改为新管理员的 Telegram 数字 ID
2. 重启 Bot 进程（`docker compose restart` 或 systemctl）
3. Bot 启动时检测到 `INITIAL_SUPER_ADMIN_ID` 与当前数据库中的超级管理员不一致 → 强制覆盖：原超级管理员降级为副管理员（保留），新 ID 成为超级管理员
4. 该覆盖动作写入 audit_logs，actor 标记为 'env_override'

### 4.5 可维护性
- 模块化分层：`handlers / services / repositories / models / keyboards`
- 完整的类型注解
- 关键业务逻辑覆盖单元测试（pytest）
- Inline Keyboard 工厂集中管理，避免按钮回调签名散落

## 6. 状态机：申请生命周期

```
[/start] ─► wizard_step=0
            │
            ▼
[选择邀请人] ─► wizard_step=1 ──[« 取消]──► cancelled
            │
            ▼

            │
            ▼

            │
            ▼  [✅ 确认]
        pending ───► 路由审核：
            │
            ├─ inviter.review_mode='self'    ─► 通知该邀请人
            └─ inviter.review_mode='admin_delegated' ─► 通知所有管理员（行锁）
                    │
                    ├─[✅ 通过]──► approved ──► 生成一次性链接 ──► 推送申请人 + 日志频道
                    │                                                    │
                    │                                                    ▼
                    │                                            [chat_member 监听]
                    │                                                    │
                    │                                          ┌─────────┴─────────┐
                    │                                          ▼                   ▼
                    │                                       used               anomaly (告警)
                    │                                          │
                    │                                          ▼
                    │                                       日志频道
                    │
                    └─[❌ 拒绝]──► rejected ──► 通知申请人（附原因） + 日志频道
```

## 7. 边界与异常处理

| 场景 | 处理方式 |
|------|---------|
| 用户重复 `/start` 且已有 pending 申请 | 按钮提示"已有进行中申请，请选择"：`[📋 查看进度]` / `[🔄 取消并重新申请]`。同一申请人**严格不允许同时向多个邀请人申请** |
| 用户被拒绝后立刻重新申请 | **允许**，无冷却期；可直接 /start 重新走流程 |
| 超级管理员账号丢失，需通过 .env 换人 | SSH 改 `INITIAL_SUPER_ADMIN_ID` → 重启 Bot → 自动覆盖；原超级管理员降级为副管理员保留；写 audit_logs |
| 副管理员尝试访问 `[⚙️ 系统配置]` | UI 层不渲染按钮 + callback 校验拦截 + 审计日志 + Bot 回复"无权限" |
| 超级管理员修改邀请链接有效期 | 仅影响后续新签发，存量链接保留原 expire_date |
| 出击报告 forwardMessage 失败（原消息被删/Bot 被踢） | attack_report_forwards.status='failed' + 错误信息；日志频道告警；主流程不阻塞 |
| 用户在 wizard 中乱发消息 | Bot 忽略文本输入，回复"请使用下方按钮操作"并重新展示当前步骤按钮 |
| 用户在材料提交步骤试图发送媒体组 | Bot 拒绝并提示"请单张提交，不要打包发送"，当前步骤保持原地等待，不消耗进度 |
| 用户提交错误类型（应发图片却发文字 / 反之） | 提示当前步骤所需类型，要求重新发送，不前进到下一步 |
| 出击报告超过媒体组 caption 上限（1024 字符） | 审核消息降级为三条：媒体组（无 caption）+ 出击报告独立消息 + 申请人信息+按钮 |
| 邀请人离开 Telegram / 被删除 | 该邀请人下的 pending 申请改由所有管理员统一兜底审核（视为代审型） |
| 代审型邀请人无 telegram_user_id | review_mode 保持 admin_delegated，仅作为挂名邀请人展示 |
| 邀请链接 24 小时未使用 | 标记 expired，日志频道推送，不重新签发；用户须重新申请 |
| 审核通过后 API 调用失败 | 重试 3 次仍失败 → 通知所有管理员 + 日志频道告警，状态保持 approved-pending-link 待人工处理 |
| 多个管理员同时点击同一代审申请 | 数据库行锁：先到先得，后到者按钮失效并提示"已被 @xxx 处理" |
| 日志频道未配置 | 仅在内部日志中记录，不影响主流程；管理员面板顶部提示「⚠️ 日志频道未配置」 |
| 出击报告频道未配置 | 审核通过流程正常完成，出击报告不转发；管理员面板顶部提示「⚠️ 出击报告频道未配置」；写一条 status='skipped_no_channel' 记录 |
| 申请不包含出击报告材料项 | 静默跳过转发，写一条 status='skipped_no_report' 记录 |
| 出击报告频道转发失败（API 错误/被踢出频道） | 重试 3 次（指数退避）；仍失败则记录 status='failed' + 错误信息，并在日志频道推送告警 + @所有管理员；主流程不阻塞（链接已正常发给申请人） |
| 出击报告超过 4096 字符 | 自动拆分多条发送，首条带头部信息（申请 ID/邀请人/申请人/时间），后续条目仅含正文续段 |
| 出击报告频道被更换后回溯旧记录 | attack_report_forwards 表 channel_id 字段为写入时快照，不随当前配置变化 |
| 用户用**原账号**尝试回群密钥 | 第 3 步同 ID 校验拦截，返回友好提示，写入 recovery_key_attempts (result='same_id')，日志频道异常告警 |
| 回群密钥被截图泄露 | 1) 同 ID 拦截已防原账号自刷；2) 异常多账号尝试触发频率限制；3) 管理员可在 §3.3.10 一键撤销/重置 |
| 密钥关联的邀请人已停用/被删除 | 拒绝使用，提示「关联邀请人已停用，请联系管理员」；管理员可在密钥详情页改绑到其他 active 邀请人 |
| 用密钥获得链接后转发给第三方入群 | chat_member 监听到实际入群人 ID ≠ 本次密钥使用人 ID → 标记 anomaly + 日志频道告警 + 通知所有管理员 |
| 同一密钥短时间内大量失败尝试 | 1 小时内失败 ≥ 5 次自动锁定，需管理员重置；同时推送日志频道告警 |
| 用户提交格式正确但不存在的密钥 | 统一返回「密钥无效、已使用或已撤销」（不区分原因，防探测）；写入 recovery_key_attempts |
| 原账号为目标群组的管理员/群主 | 跳过自动踢/封，cleanup_action='skip_admin'；日志频道告警 + 通知所有管理员手动决断 |
| Bot 在目标群缺少封禁/踢出权限 | 清理失败 cleanup_action='failed_no_permission'；不阻塞新链接发送；日志频道告警 + 通知所有管理员 |
| 无法判定原账号是否已注销（getChat 抛 timeout 等非确定性错误） | cleanup_old_account_status='unknown'，安全起见**按"正常账号"处理**（仅在群内时踢出，不做封禁） |
| 原账号被踢后通过其他邀请人重新申请 | 走正常申请流程；前端无特殊提示。若需阻断须管理员主动拉黑 |
| 用户主动重置密钥触发 1 次/天 限制 | UI 提示"今日已重置过密钥，请明日再试"，不消耗任何配额 |
| 媒体组与文本消息发送顺序错乱 | 强制按 `sendMediaGroup` → `sendMessage` 顺序，前者失败则降级为多张单图 |
| 黑名单用户尝试 `/start` | 静默拒绝，记录尝试日志 |
| 副管理员尝试添加/移除管理员 | UI 层不显示按钮 + 服务端 callback 校验拦截 + 写审计日志 + Bot 回复"无权限" |
| 超级管理员尝试移除自己 | 不可达（无此按钮 / 无此 callback）；如需退出请先「🔄 转让身份」 |
| 超级管理员转让后又反悔 | 转让一经确认即生效，原超级管理员变为副管理员；如需复位需新超级管理员再次转让 |

## 8.5 报销系统（v2 新增）

> v1.0.0 完成后增补的功能模块。用户审核通过入群后，可凭同一套材料发起报销，由管理员审核 → 输入支付宝口令红包 → Bot 转发付款。

### 8.5.1 设计目标

- 让已入群成员凭"约课记录 / 上课手势 / 出击报告"申请活动报销
- 全程内联按钮 + 多步引导，与入群审核风格一致
- 强约束：成员资格 + 冷却时间 + 月预算池 三层限流
- 完整审计：每次申请、每次审核、每次付款都落库 + 日志频道留痕
- 周报 / 月报自动私聊推送给所有超级管理员

### 8.5.2 角色权责（在主体系上的扩展）

| 角色 | 新增权责 |
|------|---------|
| **申请人** | 已通过入群审核者可发起报销；提交材料；接收口令红包 |
| **审核者**（任一管理员） | 双消息推送（含媒体组+caption）→ 通过/拒绝 → 通过后**输入支付宝口令红包文本** → Bot 转发给申请人 |
| **超级管理员** | 全权配置：固定金额、月预算、冷却天数、重置日、资格群/频道列表；按用户覆盖冷却；接收周报/月报 |

### 8.5.3 用户侧：发起报销

#### 8.5.3.1 入口

- `/reimburse` 命令 或 在 `/start` 欢迎卡片增加 `[💰 申请报销]` 按钮
- 入口立即做 **3 层预校验**：
  1. **入群资格**：`applications.status='approved'` 至少 1 条
  2. **群组/频道资格**：在**所有**已配置 `eligibility_chats` 中均为 member/admin/creator/restricted
  3. **冷却时间**：上次审核通过的 `reviewed_at` 距今 ≥ `cooldown_days`（默认 7，可全局调整或按用户覆盖）
  4. **月预算**：当前 `monthly_remaining > fixed_amount`（不能扣到负）

任一失败 → 友好提示文案 + 终止，**不进 wizard**。

#### 8.5.3.2 报销 wizard

引导式逐项提交，与入群 wizard 完全一致的规则：
- 步骤 1：约课记录（单张图，禁媒体组）
- 步骤 2：上课手势（单张图）
- 步骤 3：出击报告（文本）
- 步骤 4：预览 → `[✅ 确认提交]` / `[✏️ 重新提交]` / `[« 上一步]` / `[❌ 取消]`

提交后 `reimbursement_requests.status='pending'`，触发管理员侧推送。

### 8.5.4 管理员侧：审核 + 付款

#### 8.5.4.1 推送（与 §3.2.1 双消息同构）

- 消息 ①：媒体组（约课记录 + 上课手势），首图 caption = 出击报告
- 消息 ②：申请人信息 + 金额 + 历史报销次数 + `[✅ 通过] [❌ 拒绝] [👁 重发审核材料]`
- 报告超长 → 降级三消息
- 行锁防多管理员并发审核（同 §3.3.5 代审型）

#### 8.5.4.2 通过流程

1. 管理员点 `[✅ 通过]`
2. 检查月预算 `>= fixed_amount`；不足 → 该申请改 `rejected`（原因 = "本月预算已用尽，已退回"）+ 推送日志 + 通知申请人
3. 月预算扣减 `fixed_amount`，`status='approved'` + `reviewed_at`
4. Bot 私聊审核者："✅ 已批准，请发送支付宝口令红包文本（**5 分钟内有效**）"
5. 审核者发文本 → Bot 校验非空 → 保存到 `reimbursement_requests.alipay_code_text` + `paid_by_telegram_id` + `paid_at`
6. Bot 把口令文本转发给申请人："🎁 您的报销已批准，请兑换"
7. `status='paid'`，编辑审核消息 ② 为 `✅ 已通过 + 已付款 by @xxx · HH:MM`
8. 日志频道：`💰 报销已发放` 卡片

**异常分支：**
- 5 分钟超时未发口令 → `status='approved'`（保留），管理员可后续 `/admin → 报销管理 → 已批准待付款` 补发
- 管理员发的不是文本（如图片）→ Bot 提示"请发送文本口令"，状态保留

#### 8.5.4.3 拒绝流程

- 与入群审核同（§3.2.2）：`[✏️ 填写原因] / [⏩ 跳过]`
- 拒绝**不扣预算**
- 通知申请人 + 日志频道 `❌ 报销拒绝`

### 8.5.5 报销池（月预算）

| 配置项 | 默认 | 说明 |
|--------|:----:|------|
| `reimbursement_fixed_amount_cents` | 0 | 每次报销固定金额（分）。0 表示报销功能未启用 |
| `reimbursement_monthly_budget_cents` | 0 | 当月预算（分）。0 = 无预算（即不开启月限） |
| `reimbursement_monthly_remaining_cents` | = 月预算 | 实时余额；扣减式 |
| `reimbursement_budget_reset_day` | 1 | 每月第几日 00:00 重置（1–28） |
| `reimbursement_global_enabled` | false | 总开关，默认关闭，需超管显式打开 |

**扣减时机**：管理员点 `[✅ 通过]` 那一刻扣减；拒绝不扣，月底未发放的也不退（账面已花 = 已发放，与"批准但未付款"的处理简化合一）。

**重置**：每天 00:00 由 JobQueue 检查；当日是 `reset_day` 则把 `monthly_remaining` 重置为 `monthly_budget`。

### 8.5.6 冷却时间（rate limiting）

| 配置项 | 默认 | 说明 |
|--------|:----:|------|
| `reimbursement_default_cooldown_days` | 7 | 全局默认冷却天数；下次报销须距上次 `reviewed_at` ≥ N 天 |
| `reimbursement_user_overrides`（表）| – | 单用户覆盖：`telegram_user_id → cooldown_days` |

**起算点**：上一次 `status='approved'` 或 `status='paid'` 的 `reviewed_at`；rejected 的不计入冷却。

### 8.5.7 资格校验（成员身份）

#### 8.5.7.1 资格群组/频道列表
- 新表 `eligibility_chats`：管理员通过 `/admin → 报销管理 → 资格列表` 添加（转发消息识别 chat_id，同 §3.3.2）
- 必须**全部都是 active**，且申请人**在每一个里**才合格
- 校验时机：用户每次发 `/reimburse` 时即查

#### 8.5.7.2 校验实现
- 对每个 active 配置项调 `getChatMember(chat_id, applicant_user_id)`
- 任一返回 `left`/`kicked`/`not found`/异常 → 资格失败
- 缓存：成功结果 5 分钟，失败结果不缓存
- 提示文案：「您不符合报销资格，请联系管理员」（不暴露细节哪个群缺）

### 8.5.8 周报 / 月报

#### 8.5.8.1 触发
- 周报：每周一 00:05 由 JobQueue 触发，统计上周（周一 00:00 ~ 周日 23:59）
- 月报：每月 1 日 00:10 触发，统计上月

#### 8.5.8.2 内容
```

  已批准：18  已付款：18
  已拒绝：4   待审核：1
总发放：900.00 元（18 × 50 元）
本月已用：3200.00 / 5000.00 元（剩余 1800.00）

  @alice：3 次
  @bob：2 次
  @carol：2 次
  ...

```

#### 8.5.8.3 推送对象
- 私聊推送给**所有**超级管理员（不广播给副管理员）
- 失败重试 3 次后写错误日志，不阻塞 JobQueue

### 8.5.11 边界与异常处理

| 场景 | 处理方式 |
|------|---------|
| 用户未通过入群审核就发 `/reimburse` | 直接拒绝："请先完成入群申请并通过审核" |
| 资格校验失败（不在任一指定群） | "您不符合报销资格，请联系管理员"（不暴露哪个群缺） |
| 冷却时间未到 | 提示剩余天数 |
| 月预算不足（用户提交时） | 提示"本月预算已不足以支付一次报销，请月初再试" |
| 月预算不足（审核通过时） | 该申请改 rejected + 原因 "本月预算已用尽，已退回" |
| 管理员通过后 5 分钟未发口令 | 状态保持 approved，可在管理面板「待付款」补发 |
| 管理员发的不是文本 | 提示"请发送文本口令"，重试 |
| 配置 fixed_amount=0 或 monthly_budget=0 或 enabled=false | 阻止用户进入 wizard，提示"功能未启用" |
| 资格列表为空 | 视为"没有限制"还是"全员拒绝"？→ **全员拒绝**（更严谨；超管必须显式配置至少 1 个） |

### 8.5.12 安全考量

- 口令红包文本是**敏感数据**，仅审核者私聊看到 + DB 存储 + 申请人接收方私聊看到；不进入日志频道（仅卡片显示"💰 已发放，金额 50 元"）
- 月预算 / 单次金额由超级管理员配置，副管理员不可改
- 多管理员并发审核：行锁 + status 复检（同 §M3）
